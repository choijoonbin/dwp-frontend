# ⚠️ What NOT to Do (중단/방지 가이드)

> **최종 업데이트**: 2026-01-22  
> **목적**: 불필요한 리팩토링/과투자를 방지하여 안정화 구간 유지

---

## 🎯 현재 상황

**현재 단계**: 운영 안정화 구간  
**목표**: UI/UX 일관성 + 회귀 방지  
**우선순위**: 새 기능 개발 > 대규모 리팩토링

---

## ❌ 지금 단계에서 하지 말아야 할 것

### 1. Admin 모든 페이지에 FilterBar/DataTable 패턴 억지 적용 ❌

**이유**:
- Audit/Monitoring은 이미 안정적이며, 강제 통일은 **기능 변경 위험**이 큼
- 현재 구조가 잘 작동하고 있다면 "패턴 통일"만을 위해 건드리지 말 것

**허용되는 경우**:
- ✅ 새로운 기능 추가하면서 자연스럽게 패턴 적용
- ✅ 버그 수정하면서 동시에 패턴 마이그레이션
- ✅ 명확한 운영 이슈(반응형 깨짐 등)가 있을 때만

**금지되는 경우**:
- ❌ "통일성"만을 이유로 안정적인 페이지 대규모 리팩토링
- ❌ "Design System 적용률 100%"를 목표로 강제 변경

---

### 2. EditorModal 패턴 전면 교체 ❌

**이유**:
- Codes/CodeUsages의 **복잡한 form modal**은 현재 유지 결정이 맞음
- EditorModal 패턴이 모든 케이스에 맞는 것은 아님

**현재 전략**:
- ✅ 문서로 UX 규칙 고정 (Modal UX 규칙 섹션 완료됨)
- ✅ 신규 화면부터 EditorModal 적용
- ❌ 기존 복잡한 modal을 억지로 EditorModal로 교체하지 말 것

**허용되는 경우**:
- ✅ 신규 CRUD 페이지 개발 시 EditorModal 사용
- ✅ 기존 modal에 버그/UX 이슈가 있을 때 개선하면서 전환

**금지되는 경우**:
- ❌ 정상 작동하는 기존 modal을 "패턴 통일"만을 위해 교체

---

### 3. Roles/Resources 같은 페이지 대규모 UI 재디자인 ❌

**이유**:
- 지금 목표는 **"운영 회귀 방지 + 일관성"**이지, 대규모 개편이 아님
- Roles/Resources는 이미 TwoColumnLayout 적용되어 안정적

**현재 전략**:
- ✅ 현재 구조 유지
- ✅ 버그/반응형 이슈만 수정
- ❌ "더 예쁘게"를 이유로 대규모 UI 변경하지 말 것

**허용되는 경우**:
- ✅ 명확한 UX 개선 요구사항이 있을 때 (PM/디자이너 승인)
- ✅ 운영 중 발견된 문제를 해결하면서 자연스럽게 개선

**금지되는 경우**:
- ❌ "더 Modern하게" 같은 주관적 이유로 재디자인
- ❌ Sidebar/Header 같은 전역 레이아웃 대규모 변경

---

### 4. shadcn/radix/lucide를 도입해서 해결하는 방식 ❌

**이유**:
- 이미 **ESLint 규칙 위반**이며, Design System 정책과 충돌
- MUI v5가 프로젝트 표준이며, 충분히 강력함

**현재 전략**:
- ✅ MUI + Iconify ONLY
- ✅ 필요한 컴포넌트는 `libs/design-system`에 직접 구현
- ❌ "shadcn이 더 좋아서"라는 이유로 도입 절대 금지

**허용되는 경우**:
- ✅ 없음 (예외 없음)

**금지되는 경우**:
- ❌ shadcn/radix/lucide 도입 (어떤 이유로든)
- ❌ "생성 코드(v0.app 등)가 요구해서"라는 핑계로 설치
- ❌ "다른 팀이 쓰고 있어서"라는 이유로 추가

**대안**:
- MUI에서 동일한 기능 찾기
- 없으면 `libs/design-system`에 직접 구현
- 정 필요하면 Tech Lead와 논의 (ADR 작성)

---

### 5. '테스트 커버리지 늘리기'를 이유로 스펙/구조 변경 ❌

**이유**:
- 지금은 **Smoke 기반 최소 회귀 방지**로 충분
- 100% 커버리지를 위해 구조를 바꾸는 것은 본말전도

**현재 전략**:
- ✅ Smoke 테스트 (페이지 로드 + 기본 요소)
- ✅ 최소 회귀 테스트 (RBAC/CodeUsage)
- ✅ A11y 기본 검증 (focus/ESC/Tab)
- ❌ 100% 커버리지를 목표로 하지 말 것

**허용되는 경우**:
- ✅ 운영 중 발견된 회귀 버그를 테스트로 재현
- ✅ 신규 기능 추가 시 해당 기능만 테스트 추가

**금지되는 경우**:
- ❌ "테스트하기 쉽게"를 이유로 코드 구조 변경
- ❌ "커버리지 80% 달성"을 목표로 대규모 테스트 추가
- ❌ Unit 테스트를 위해 컴포넌트를 과도하게 분리

---

### 6. Storybook 지금 당장 도입 ❌

**이유**:
- Storybook은 **중장기 투자**이지, 지금 긴급 우선순위 아님
- 현재 팀 규모/컴포넌트 개수에서는 문서로 충분

**도입 시기 판단 기준**:
- ✅ Remote 앱이 3개 이상으로 늘어났을 때
- ✅ 공통 컴포넌트가 15개 이상으로 늘어났을 때
- ✅ 디자이너/기획과 "UI 합의"가 필요한 상황일 때

**현재 전략**:
- ✅ `docs/essentials/DESIGN_SYSTEM.md`로 문서화
- ✅ "도입 준비 상태로 유지" (미래에 필요하면 빠르게 도입)
- ❌ 지금 당장 Storybook 설정/스토리 작성하지 말 것

**허용되는 경우**:
- ✅ PM/Tech Lead가 명확히 승인했을 때
- ✅ 팀이 3명 이상으로 늘어나고 컴포넌트 충돌이 잦을 때

**금지되는 경우**:
- ❌ "있으면 좋겠어서" 도입
- ❌ "다른 프로젝트가 쓰고 있어서" 도입

---

### 7. 전역 상태 관리 라이브러리 추가 ❌

**이유**:
- 현재 **Zustand + TanStack Query**로 충분히 해결됨
- Redux/Recoil/Jotai 등 추가는 복잡도만 증가

**현재 전략**:
- ✅ 전역 UI 상태: Zustand
- ✅ 서버 데이터: TanStack Query
- ✅ URL 상태: React Router (searchParams)
- ❌ 추가 상태 관리 라이브러리 도입 금지

**허용되는 경우**:
- ✅ Zustand/TanStack Query로 해결 불가능한 명확한 케이스 (ADR 필수)

**금지되는 경우**:
- ❌ "Redux가 더 익숙해서"
- ❌ "Zustand가 부족해 보여서"

---

### 8. 대규모 폴더 구조 리팩토링 ❌

**이유**:
- 현재 구조가 **명확하고 안정적**
- 폴더 구조 변경은 import 경로 대규모 수정 = 높은 리스크

**현재 전략**:
- ✅ 현재 구조 유지 (`pages/<feature>/` 구조)
- ✅ 신규 feature만 표준 구조 적용
- ❌ 기존 폴더 구조 대규모 변경 금지

**허용되는 경우**:
- ✅ 명확한 문제(순환 참조 등)가 있을 때
- ✅ Tech Lead 승인 + ADR 작성

**금지되는 경우**:
- ❌ "더 예쁜 구조"를 위한 리팩토링
- ❌ "다른 프로젝트 구조가 좋아 보여서"

---

## ✅ 지금 단계에서 해야 하는 것 (요약)

1. **새 기능 개발** (Admin 페이지 추가, 신규 Remote 앱 등)
2. **버그 수정** (운영 중 발견된 이슈)
3. **반응형 이슈 해결** (모바일에서 깨지는 부분)
4. **접근성 개선** (명확한 UX 이슈)
5. **문서 업데이트** (팀 온보딩 개선)

## 📋 의사결정 프로세스

### "이 작업을 지금 해야 할까?" 판단 기준

```
1. 이 작업이 운영 이슈를 해결하는가?
   → YES: 진행 ✅
   → NO: 2번으로

2. 이 작업이 신규 기능 개발인가?
   → YES: 진행 ✅
   → NO: 3번으로

3. 이 작업이 명확한 PM/디자이너 요구사항인가?
   → YES: 진행 ✅
   → NO: 4번으로

4. 이 작업이 "일관성/패턴/리팩토링"만을 위한 것인가?
   → YES: ❌ 보류 (다음 분기로 미룸)
   → NO: Tech Lead와 논의
```

## 🚨 예외 요청 절차

만약 위 "금지 사항"을 꼭 해야 한다고 판단되면:

1. **요청서 작성** (Slack/PR 코멘트)
   ```markdown
   ### 예외 작업 요청
   - 작업명: [작업 설명]
   - 금지 사항: [어떤 금지 사항에 해당하는지]
   - 사유: [왜 지금 꼭 해야 하는지 - 구체적으로]
   - 대안 검토: [다른 방법은 없는지]
   - 영향 범위: [어디까지 영향을 미치는지]
   - 예상 소요 시간: [개발 + 테스트 포함]
   ```

2. **Tech Lead 승인** (필수)

3. **ADR 작성** (Architecture Decision Record)

4. **팀 공지** (Slack 등)

5. **작업 진행**
